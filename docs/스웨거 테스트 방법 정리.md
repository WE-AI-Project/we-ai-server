`http://localhost:8080/swagger-ui/index.html`

## 1. 시작 전 체크리스트

1. 백엔드 앱이 `8080` 포트에서 실행 중인지 확인합니다.
2. Swagger UI에 접속합니다.
   - `http://localhost:8080/swagger-ui/index.html`
3. 로그인 후 받은 `accessToken`을 Swagger 우측 상단 `Authorize`에 넣습니다.
   - 형식: `Bearer {accessToken}`
4. Swagger 요청 본문은 반드시 `순수 JSON 객체`로 붙여 넣습니다.
   - 바깥쪽 큰따옴표가 있으면 안 됩니다.
   - `\n`, `\"` 가 그대로 보이면 잘못 붙여 넣은 상태입니다.
5. Windows 경로는 `C:/...` 형태를 권장합니다.
   - 예: `C:/WE_AI/schedule-api-test`

## 2. 먼저 메모해둘 변수

테스트하면서 아래 값들을 메모해두면 이후 API를 연속으로 성공시키기 쉽습니다.

| 변수 | 어디서 얻는지 |
|---|---|
| `{accessToken}` | 로그인 성공 응답 |
| `{refreshToken}` | 로그인 성공 응답 |
| `{projectId}` | 프로젝트 생성 응답의 `data.projectId` |
| `{projectCode}` | 프로젝트 생성 응답의 `data.projectCode` |
| `{memberId}` | 멤버 목록 응답의 `data.members[].projectMemberId` |
| `{memberUserId}` | 멤버 목록 응답의 `data.members[].userId` |
| `{techStackId}` | 기술 스택 생성 또는 조회 응답의 `data.techStacks[].techStackId` |
| `{scheduleId}` | 일정 생성 또는 조회 응답의 `data.schedules[].scheduleId` |

중요:

- `assigneeId`는 `project_members.project_member_id`가 아니라 `users.id`입니다.
- 즉, 일정 담당자 지정 시에는 멤버 목록에서 `projectMemberId`가 아니라 `userId`를 넣어야 합니다.

## 3. 추천 테스트 계정 구성

모든 프로젝트 API를 자연스럽게 테스트하려면 계정을 2개 준비하는 것이 가장 좋습니다.

1. 리더 계정
   - 프로젝트 생성
   - 기술 스택/일정/멤버 관리
2. 멤버 계정
   - 참여 코드로 프로젝트 참여
   - 멤버 상세/역할/부서 변경 테스트 대상

혼자 테스트해도 되지만, `프로젝트 참여`, `멤버 역할 변경`, `멤버 부서 변경`은 두 번째 계정이 있으면 훨씬 안정적으로 확인할 수 있습니다.

멤버 계정 생성 예시:

```json

  "username": "member0707",
  "name": "멤버사용자",
  "email": "member0707@example.com",
  "password": "member1234!"
```

리더 로그인 예시:


### 4-2. 프로젝트 기본 흐름

1. JSON 앞뒤에 큰따옴표가 없는지
2. 줄바꿈 문자가 `\n`으로 들어가 있지 않은지

### 4-6. 프로젝트 대시보드 세부 API 흐름

이 구간은 프로젝트, 멤버, 일정, 기술 스택이 조금이라도 쌓인 뒤에 보는 것이 가장 좋습니다.

  - `/api/v1/projects/{projectId}/dashboard/activities?limit=5`

#### 28. `GET /api/v1/projects/{projectId}/dashboard/progress`

- 목적: 일정 상태별 통계 확인
- 일정이 0개면 `progressRate`가 `0`으로 나와도 정상입니다.

#### 29. `GET /api/v1/projects/{projectId}/dashboard/departments`

- 목적: 부서별 멤버 수, 일정 수, 진행률 확인

#### 30. `GET /api/v1/projects/{projectId}/dashboard/milestones`

- 기본 호출:
  - `/api/v1/projects/{projectId}/dashboard/milestones`
- 상태 필터:
  - `/api/v1/projects/{projectId}/dashboard/milestones?status=IN_PROGRESS`

이 API는 조회 전용이므로, 데이터가 없으면 빈 배열이 나와도 정상입니다.

마일스톤을 실제로 넣고 테스트하고 싶으면 아래 SQL을 사용합니다.

```sql
INSERT INTO project_milestones (
  created_at,
  updated_at,
  project_id,
  title,
  description,
  start_date,
  end_date,
  status,
  progress_rate
) VALUES (
  NOW(6),
  NOW(6),
  {projectId},
  '1차 마일스톤',
  '핵심 API 구현 완료',
  '2026-06-09',
  '2026-06-16',
  'IN_PROGRESS',
  55
);
```

확인 SQL:

```sql
SELECT milestone_id, project_id, title, status, progress_rate, start_date, end_date
FROM project_milestones
WHERE project_id = {projectId}
ORDER BY start_date ASC;
```

## 5. AI API 테스트

AI API는 일반 CRUD API와 다르게 로컬 AI 환경 준비 여부에 따라 성공 여부가 달라질 수 있습니다.

권장:

1. 일반 프로젝트 API를 먼저 전부 확인합니다.
2. 그 다음 AI API를 별도로 테스트합니다.

#### 37. `POST /api/v1/ai/qa`

```json
{
  "diff": "diff --git a/src/main/java/com/example/UserService.java b/src/main/java/com/example/UserService.java\n@@\n- return userRepository.findById(id).get();\n+ return userRepository.findById(id).orElse(null);\n"
}
```

#### 38. `POST /api/v1/ai/debate`

```json
{
  "query": "프로젝트 일정 관리 API를 더 안정적으로 운영하려면 어떤 구조가 좋을까요?"
}
```

#### 39. `POST /api/v1/ai/chat`

```json
{
  "question": "프로젝트에서 인증 코드는 어떤 방식으로 검증되나요?"
}
```

AI API가 실패하면 아래를 먼저 확인합니다.

1. Ollama 또는 AI 모델 서버가 떠 있는지
2. ChromaDB가 떠 있는지
3. 관련 `.env` 값이 정상인지
4. 서버 시작 로그에 AI Bean 초기화 오류가 없는지

## 7. 자주 막히는 포인트

### 1. `400 Request body could not be parsed.`

원인:

- JSON 전체가 문자열로 들어감
- 줄바꿈이 `\n` 문자 그대로 들어감
- Windows 경로의 역슬래시 때문에 Swagger 예시가 꼬임

해결:

- 직접 JSON 객체를 다시 붙여 넣습니다.
- 경로는 `C:/...` 로 바꿉니다.

### 2. `401 Unauthorized`

원인:

- `Authorize`에 토큰이 없거나 만료됨

해결:

1. 다시 로그인합니다.
2. 새 `accessToken`으로 `Authorize`를 다시 설정합니다.

### 3. `403 PROJECT_ACCESS_DENIED`

원인:

- 해당 프로젝트의 ACTIVE 멤버가 아님

해결:

- 멤버 계정으로 `POST /projects/join`을 먼저 수행합니다.

### 4. `409` 중복 오류

대표 예시:

- 이미 가입한 이메일
- 이미 존재하는 기술 스택 이름과 카테고리 조합

해결:

- 이메일, 기술 스택 이름, 프로젝트명 등을 조금 바꿔서 다시 테스트합니다.
